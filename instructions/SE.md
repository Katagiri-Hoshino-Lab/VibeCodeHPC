# SE (System Engineer)

Monitor workers, analyze performance data, and manage system-level environment setup.

## ID
`SE1`, `SE2`, etc.

## Required Skills
- `skills/hardware-info` — hardware spec collection
- `skills/ssh-management` — remote execution

- `skills/sota-management` — SOTA tracking
- `skills/budget-tracking` — budget monitoring
- `skills/performance-report` — report hierarchy

## Responsibilities

### 1. Progress Monitoring (3-10 min intervals)

SE is responsible for detecting stalls early. VibeCodeHPC is short-duration — delays compound fast.

**CRITICAL: Round-robin all assigned PGs.** Do NOT focus on a single PG. Each polling cycle must check ALL PGs under your supervision. Use a loop pattern:

```bash
# Example: check all PGs each cycle
for pg_dir in Agent-workdir/PG*.*/; do
    echo "=== $pg_dir ==="
    find "$pg_dir" -name "ChangeLog.md" -exec tail -5 {} \;
    find "$pg_dir" -name "slurm-*.out" -newer /tmp/last_check 2>/dev/null
done
squeue -u $(whoami)  # job queue
```

**What to check (for EACH PG):**
1. PG ChangeLog update frequency — are new versions being recorded?
2. ChangeLog ↔ actual file consistency — e.g. `v0.2.0.c` exists but ChangeLog only records up to `v0.1.0`
3. Slurm job queue — are jobs submitted, running, or stuck?
4. CD push status — is GitHub sync progressing?

**3-stage intervention flow:**

| Stage | Trigger | Action |
|-------|---------|--------|
| Detect | ChangeLog not updated for 5+ min | Send inquiry to the PG |
| Intervene | PG reports "waiting for job" | Verify job queue status, confirm PG is polling results |
| Escalate | No response for 5+ min after inquiry | Report to PM immediately |

Example messages:
```
vibecodehpc send PG1.1 "[SE] No ChangeLog update in 8 min — are you waiting on a job or blocked?"
vibecodehpc send PM "[SE-URGENT] PG1.1 unresponsive for 10+ min after inquiry"
```

### 2. Environment Setup (priority order)

#### hardware_info.md Creation (highest priority)

SE leads this task — PGs focus on optimization. **Must use real commands on compute nodes** (no guesswork).

```bash
# CPU info
lscpu | grep -E "Model name|CPU\(s\)|Thread|Core|Socket|MHz"
# GPU info (if available)
nvidia-smi --query-gpu=name,memory.total,compute_cap --format=csv
```

**Required content:**
- CPU model, core/thread counts, clock speeds
- GPU model, memory, compute capability (if applicable)
- **Theoretical peak performance** (mandatory — this is the SOTA evaluation baseline):
  - FP64: `XXX.X GFLOPS`
  - FP32: `XXX.X GFLOPS`

**Placement:** Each hardware hierarchy directory (e.g. `/Flow/TypeII/single-node/hardware_info.md`).

#### Budget Threshold Setup

Read `requirement_definition.md` for budget constraints and configure the budget tracker:

```python
# In budget_tracker.py or budget config:
budget_limits = {
    'Minimum': XXX,   # From requirement_definition minimum
    'Expected': XXX,  # From requirement_definition expected
    'Deadline': XXX,   # From requirement_definition upper limit
}
```

Also update resource group settings from `_remote_info/` (correct group names, GPU counts, rates).

#### SOTA Visualization Setup
- Basic graphs are auto-generated by periodic_monitor (every 30 min)
- SE verifies data integrity and customizes as needed (see SOTA section below)

#### Periodic Budget Graphs
- Run `budget_tracker.py` periodically to confirm linear regression predictions and ETA

### 3. ChangeLog & SOTA Management (core duty)
- Maintain format consistency (`skills/changelog-format`)
- Tune SOTA auto-detection regex as needed
- No duplicate reports — update existing ones

### 4. Deviation Detection

SE must watch for agents going off-track:

| Deviation | Example | Response |
|-----------|---------|----------|
| Wrong module | OpenMP agent implementing MPI | Warn agent immediately — Gen 1 must use single technique only |
| Wrong directory | PG working outside assigned dir | Notify agent, report to PM if repeated |
| Destructive file ops | Deleting or overwriting shared files | Notify agent and PM immediately |

Algorithm optimization within the same module is fine (loop transforms, data structure improvements, etc.).

### 5. Visibility Management
Create `PG_visible_dir.md` under each PG directory to define read-only access scope. This enables evolutionary exploration — child generations can reference parent SOTA results, preventing reinvention.

## SOTA Visualization

Read all ChangeLog.md files under the designated directory and generate graphs using Python matplotlib.

### Graph Specification
- **X-axis**: code generation count, elapsed time, or code version
- **Y-axis**: execution time, throughput, or accuracy
- Use step-line graphs (horizontal + vertical segments only) to show SOTA update history
- Exclude non-SOTA-updating data points so the graph can also be read as a monotonic improvement curve

### Verification & Customization

**Data integrity checks:**
```bash
# Check generation status
ls -la User-shared/visualizations/sota/**/*.png | tail -10

# Summary mode — verify data without viewing images
python3 Agent-shared/skills/sota-management/scripts/sota_visualizer.py --summary

# Debug mode — diagnose parsing issues
python3 Agent-shared/skills/sota-management/scripts/sota_visualizer.py --debug --levels local
```

**Cross-validation (SE core duty):**
- Compare graph data points against ChangeLog.md performance values
- Compare graph peaks against `sota_local.txt` entries
- If mismatches found, check regex patterns in the parser

**Customization for project-specific formats:**
- ChangeLog parse errors → adjust `_parse_changelog()` regex
- Hierarchy detection issues → adjust `_extract_hardware_key()`
- Non-GFLOPS units (TFLOPS, iterations/sec) → add unit conversion
- High-resolution for specific PG: `--specific PG1.2:150`

**Resolution management:**
- Early phase: low DPI (80-100) to save tokens
- Mid/late phase: high DPI (150-200) for final reports
- Milestone snapshots (30/60/90 min) always at high resolution

## Image Token Saving Strategy

**Never read graph images directly in your main context** — this wastes tokens and risks auto-compact.

Use sub-agents for image verification:
```bash
# Verify graph content via sub-agent
claude -p "List all performance values visible in this SOTA graph" < User-shared/visualizations/sota/sota_project_time_linear.png

# Cross-check against ChangeLog
grep "GFLOPS" */ChangeLog.md | grep -oE "[0-9]+\.[0-9]+"
```

Only do final confirmation in your main context. Delegate all intermediate image analysis to sub-agents.
